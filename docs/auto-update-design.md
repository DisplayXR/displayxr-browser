# Auto-update — delivering security patches to installed browsers

## Why this is a 1.0 blocker

The [security-rebase pipeline](security-rebase-automation.md) makes patched builds *exist*. This
makes them **arrive**. Without it the security commitment is hollow: today a user who installed
0.1.4 gets a "new version available → download" prompt they will ignore, so weeks later they are
still running an unpatched browser while we tell them it's maintained. **That is the difference
between "we ship patches" and "users are patched," and only the second one earns a non-Preview 1.0.**

Tracked as #40; the pipeline side is #33/#38.

## Constraints that shape the design

- **Small team.** Whatever we adopt has to survive us not touching it for months.
- **Windows-only** (current browser scope), NSIS installer, already Authenticode-signed by the
  release flow (validated on 0.1.4).
- **We already have** a release feed (GitHub Releases), signed installers, and a launch-time version
  check — this is an upgrade of an existing path, not a greenfield system.
- **Cadence** is the *security* cadence (~1–2 weeks), not the milestone cadence.
- **Trust posture:** we just removed every long-lived credential from publishing (OIDC/trusted
  publishing). An updater that installs unverified bytes would undo that.

## Options considered

| Option | Verdict |
|---|---|
| **Omaha** (Chrome's classic updater) | Rejected for now — needs Omaha-protocol server infra + a COM/Windows-service install; a standing operational commitment far larger than the thing it updates. |
| **`chrome/updater`** (the in-tree modern updater we already compile) | Attractive — maintained by Google, already in our source. But it speaks the Omaha v3 protocol and still needs a compatible server, so it carries Omaha's server cost without Omaha's maturity for third parties. **Revisit** if we ever want delta updates + enterprise policy for free. |
| **WinSparkle / Squirrel.Windows** | Squirrel is effectively unmaintained and .NET-shaped. WinSparkle is solid but assumes an app-shaped update, and we'd still bolt on our own trust + rollout logic. |
| **Purpose-built updater over a signed feed** ✅ | **Recommended.** Reuses signing + installer + release hosting we already operate; no server to run; a bounded amount of code we fully control. |

## The design

### 1. The feed
A small JSON document at a **stable URL** (GitHub Pages or a fixed Release asset), regenerated by the
publish step (#38 — "bump the update feed"):

```json
{
  "schema": 1,
  "channel": "preview",
  "latest": { "version": "0.1.5", "chromium": "150.0.7871.129",
              "url": "https://github.com/.../DisplayXR-Browser-Preview-Setup-0.1.5.exe",
              "sha256": "…", "size": 365009120, "released": "2026-07-20T…Z",
              "security": true },
  "minimum": "0.1.3",
  "rollout": 100
}
```

- `security: true` marks a release that carries CVE fixes — the updater treats these as urgent
  (see cadence below).
- `minimum` lets us force-upgrade builds we consider unsafe to keep running.
- `rollout` is the percentage-gate that later lets #38 go **full auto-publish** (see §4).

### 2. The check
The existing launch-time version check becomes a **background check on launch + every 6h**. It is a
plain HTTPS GET of the feed — no telemetry, no user identifier beyond a random install GUID used
only for rollout bucketing (§4).

### 3. Download → verify → install (the trust chain)
This is the security-critical part and must be conservative:

1. Download the installer to a temp path.
2. **Verify SHA-256** against the feed value.
3. **Verify Authenticode**: signature status `Valid` **and** the signer is our cert (subject contains
   `Leia, Inc.`). This is the same gate the release flow already asserts.
4. Only if both pass, run the NSIS installer **silently** (`/S`).
5. Apply on the next browser restart, Chrome-style ("Relaunch to finish updating"). For a
   `security: true` release, nag harder and relaunch on idle if the user leaves it running for days.

**Never** execute a downloaded artifact that fails either check — fail closed, log, retry next cycle.
The feed is fetched over HTTPS; the *authority* for "is this safe to run" is the Authenticode
signature + hash, not the transport, so a compromised feed host still cannot ship us arbitrary code.

### 4. Staged rollout (this is what unblocks #38)
Each install stores a random GUID once. `bucket = crc32(guid) % 100`; take the update only if
`bucket < rollout`. Publishing then becomes: feed at `rollout: 10` → watch → raise to `100`.

That gives us a **blast-radius limit we do not have today**, which is precisely the reason #38
currently requires a human to promote the feed flip. **Once staged rollout ships, #38 can go full
auto-publish** — the human checkpoint exists only because a bad build would otherwise reach 100% of
users at once. Rollback = republish the feed pointing `latest` at the previous good version (users
who already took the bad one need the next release; see §6).

### 5. The elephant: update size
Our installer is **~365 MB**. At a weekly security cadence that is ~1.5 GB/month/user — unacceptable
as a default. Mitigations, in order of preference:

1. **Split the cadence from the build cadence.** Build every dot-release (cheap, automated), but only
   *push* to users for `security: true` releases or monthly, whichever comes first. Most dot-releases
   can be skipped for delivery without leaving users unpatched on anything exploited.
2. **Differential updates** — Chromium ships **Zucchini** in-tree (`components/zucchini`), which is
   exactly this problem solved. A patch between adjacent dot-releases is a few MB. This is the real
   fix and is the strongest argument for revisiting `chrome/updater`, which already integrates it.
3. Compress harder / ship a stripped package.

**v1 = option 1** (cadence split, full installer). Zucchini is the obvious v2 and should be designed
for — hence `sha256` + explicit `version` in the feed, so a future `patches: [{from, to, url}]` field
is additive.

### 6. Failure + rollback
- Update fails verification or install → log, keep the current browser, retry next cycle. **Never**
  leave the user without a working browser.
- Bad build shipped → republish the feed with `latest` = previous good version and raise `minimum`
  only if the bad build is *dangerous*. Users who already took it recover on the next release.
- The updater must be resilient to a missing/malformed feed (treat as "no update", never as "force").

## Scope

**In:** Windows, the preview/stable channel, full-installer delivery, hash + Authenticode
verification, staged rollout, relaunch UX.
**Out (v1):** delta/Zucchini updates, macOS/Linux, enterprise policy/GPO, rollback-to-previous
in place, per-user vs per-machine install split beyond what the NSIS installer already does.

## Open decisions

1. **Feed host** — GitHub Pages (stable URL, easy) vs a fixed Release asset (fewer moving parts, but
   URL churn per release). Leaning Pages.
2. **Delivery cadence** — push every `security: true` release, or batch weekly? (§5 option 1.)
3. **Relaunch aggressiveness** — prompt-only, or force-relaunch after N days on a security release?
4. **Where the updater lives** — a small separate `.exe` scheduled task (updates even when the
   browser is never launched) vs in-browser only (simpler, but a browser that is never opened is
   never patched). Leaning separate task for exactly that reason.
